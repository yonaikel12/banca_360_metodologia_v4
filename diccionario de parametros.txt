# explicaicon parametros:

La configuración de los parámetros en el archivo settings.yaml constituye el "contrato metodológico" que garantiza que el pipeline no sea una "caja negra", sino un sistema de gobernanza analítica industrial

A continuación, se explica cada parámetro con su rigor estadístico y lógica de negocio:
1. Parámetros de Definición de Datos
dataset_rows: Define el volumen de la muestra. Rigor: Permite validar si el tamaño de la muestra es suficiente para obtener potencia estadística y evitar que los resultados se deban al azar

target: La variable de resultado (éxito/fracaso)
Negocio: Debe representar el evento que genera pérdida económica (ej. cierre de cuenta)

ols_target: Variable continua para el modelo de valor. Rigor: Se usa para estimar el valor intrínseco mediante regresión robusta, permitiendo identificar drivers monetarios

2. Listas de Columnas y Gobernanza de Features
feature_cols: Predictores para el modelo de clasificación. Rigor: Se deben elegir basándose en la intuición de negocio (valor, vinculación, fricción) para evitar la "mecanización ciega"

outlier_cols: Variables para detección de valores atípicos. Rigor: Los outliers pueden distorsionar la media; se deben identificar para decidir si son errores o valores legítimos que requieren estimadores robustos como la mediana

vif_cols: Variables para análisis de multicolinealidad. Rigor: Un Factor de Inflación de la Varianza (VIF) alto indica redundancia, lo que degrada la estabilidad de los coeficientes y dificulta la interpretación causal

inference_features: Subconjunto para explicar el fenómeno. Rigor: Basado en el principio de parsimonia (Navaja de Occam), buscando el modelo más simple que explique el resultado

value_model_features: Drivers para el CLV. Negocio: Incluye variables de comportamiento histórico necesarias para proyectar el flujo de caja futuro del cliente

segment_features: Variables para Clustering (K-Means). Rigor: Deben estar escaladas (puntuación Z) para que las unidades grandes no dominen injustamente el cálculo de distancia euclidiana

3. Parámetros de Validación y Modelado
test_size: Proporción de datos reservados. Rigor: Permite evaluar la capacidad de generalización del modelo en datos que no "vio" durante el entrenamiento, controlando el sobreajuste

seed: Semilla aleatoria. Rigor: Garantiza la reproducibilidad científica de los experimentos; sin ella, los resultados variarían en cada ejecución debido a la aleatoriedad del split

4. Lógica de Negocio y Activación Económica
clv_horizons: Ventanas de tiempo (meses). Negocio: Define el periodo en el que la organización espera recuperar la inversión de retención

financial_error_asymmetry: Penaliza de forma distinta los fallos. Rigor: En banca, suele ser más costoso no detectar a alguien que se va (Falso Negativo) que llamar a alguien que no pensaba irse (Falso Positivo)

contact_cost: Costo operativo de la acción CRM. Negocio: Parámetro vital para el cálculo del ROI estimado; sin él, el modelo solo da probabilidades, no rentabilidad

retention_success_rate: Probabilidad de que el cliente se quede tras la llamada. Negocio: Se basa en datos históricos para evitar proyecciones de ahorro demasiado optimistas

max_contact_share: Límite de capacidad operativa. Negocio: Restricción de presupuesto o personal; asegura que el modelo solo priorice a los clientes con mayor valor esperado neto

5. Ética y Equidad (Fairness)
sensitive_columns: Atributos sensibles (ej. género). Rigor: Permite realizar una auditoría de equidad para asegurar que el modelo no discrimine injustamente a subpoblaciones protegidas

age_bins: Categorización de la edad. Rigor: Convierte datos numéricos en categóricos para evaluar si el rendimiento del modelo es consistente a través de diferentes etapas de vida

6. Validación Temporal y Blindaje
n_splits: Número de iteraciones temporales. Rigor: En series de tiempo, la validación cruzada estándar falla; se requiere un enfoque secuencial para respetar la cronología

purge_gap_days y embargo_gap_days: Gaps de seguridad. Rigor: Evitan el data leakage (filtración de información). La purga elimina datos solapados entre train y test, y el embargo elimina datos inmediatamente posteriores para neutralizar la correlación serial
